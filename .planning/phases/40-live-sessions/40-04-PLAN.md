  <name>Task 1: Participant live session components</name>
  <files>src/components/session/ParticipantLiveView.tsx, src/components/session/ParticipantPairView.tsx</files>
  <action>
Create **src/components/session/ParticipantLiveView.tsx**:
- Props: programId: Id<'programs'>, sessions: array of session objects (from existing program page data)
- For each session, subscribe to api.course.sessionQueries.getLiveState. Find the one with status='running' (if any).
- If no running session: render nothing (return null)
- If running session found:
  - Subscribe to api.course.sessionQueries.getSessionPhases for that session
  - Look up current phase from phases array using liveState.currentPhaseId
  - Render a prominent banner at top of page:
    - Badge: "LIVE" with pulsing green dot (animate-pulse on a small green circle)
    - Phase title (large, bold)
    - LiveTimer component (reuse from src/components/session/LiveTimer.tsx) with phaseStartedAt and phaseDurationMs from liveState
    - Phase progress: "Phase X of Y" indicator
  - Below banner: render each active prompt via PromptRenderer in 'participate' mode
    - For each promptId in liveState.activePromptIds, render <PromptRenderer promptId={promptId} mode="participate" />
  - Below prompts: render ParticipantPairView for current phase
  - Presence heartbeat: use useEffect to send updatePresence mutation every 10 seconds while the component is mounted. Set status='idle' by default. When participant is interacting with a prompt input (detect via focus events or simply always send 'idle' — the PromptRenderer already handles draft saving which can update presence to 'typing').
    - Actually, for simplicity: send a heartbeat with status='idle' every 10s. The PromptRenderer's existing save/submit flow does NOT call updatePresence (that's a new concept). So add an onInteraction callback prop concept: wrap the prompt area in a div with onFocus handler that sends a 'typing' heartbeat, and onBlur sends 'idle'. This keeps PromptRenderer unchanged.
  - Visual styling: The banner should use a Card with a subtle green/blue left border or top accent to indicate "live". Use bg-primary/5 or similar soft highlight. Make it visually distinct from the rest of the page.

Create **src/components/session/ParticipantPairView.tsx**:

- Props: sessionId: Id<'programSessions'>, phaseId: Id<'sessionPhases'>
- Subscribe to api.course.sessionQueries.getMyPairs
- If no pair assignment: render nothing
- If paired: render a Card showing "Your pair" (or "Your group" for trio) with partner name(s)
  - Need to resolve Clerk userIds to display names. Query user data — check how existing components resolve userIds. The project likely has a users query or the participation data includes names. Use a simple approach: subscribe to a query that returns participant info for the program, or accept userNames as a prop from the parent.
  - For simplicity, show the partner userId(s) and let a future enhancement resolve to names. OR: query programParticipation to get user profiles. Check existing patterns in the admin program page for user name resolution.
- Card with Users icon, partner name(s), and strategy label badge.

Both components: named exports, no default exports. Use shadcn Card, Badge. Import type for type-only imports. Use ~/components/ alias.
